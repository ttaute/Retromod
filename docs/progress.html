Retromod  /  compatibility engineering

Six weeks of translating old mods onto new Minecraft.

Retromod rewrites old mod bytecode so it runs on current Minecraft across Fabric, NeoForge, and Forge. This is what shipped between 1.1.0-snapshot.2 and 1.3.0-snapshot.1: the capabilities, the longer threads of work, and a plain account of what still does not work.

The threads

Seven longer arcs of work

Each snapshot is a step; these are the lines they add up to. Skim here, then read the full history below.

Mixin translation (the 1.3.0 line)

The active line as of the current top. Where 1.2.0 could only STRIP a mixin handler whose target drifted (the feature going inert via the blocklist), 1.3.0-snapshot.1 rewrites handlers so the feature keeps working: the ValueIO save-data adapter (#48), ServerLevel-threading signature-drift re-signaturing (#69), MixinExtras injector dispatch, and a deleted-superclass owner-alias (#50). It retires four blocklist soft-fails (Darker Depths PlayerMixin, Revamped Phantoms SweepAttackMixin and PhantomMixin) and extends mixin selector remapping to all loaders and the offline CLI. Two re-signature-engine regressions were caught and fixed in review.

Forge 26.2 hosts (EventBus 6->7) and Forge->NeoForge synthetic bridges (1.2.0)

1.2.0 built the deep Forge-family work redirects can't do. Forge 26.2 (Forge 65.x) replaced EventBus 6 with EventBus 7 (BusGroup + per-type EventBus<T>), so a synthetic LegacyEventBus interface bridges the old idiom, including lambda setup listeners and priorities; verified in-game (Macaw's Bridges loads on Forge 26.2). Separately, the Forge->NeoForge construct+register spine (embedded FMLJavaModLoadingContext, the #87 registry-id thread-local bridge, extension-interface redirects, FML lifecycle/event migration, 308 renamed event classes, registry migration onto BuiltInRegistries) carries a Forge 1.21.1 content mod through scan/construct/register to the menu on NeoForge 26.2. Both rely on the SyntheticEmbedder, which embeds deleted-class bridges per-mod under a Retromod-owned package to stay JPMS split-package-safe.

MC 26.2 support across all three loaders

First-day 26.2 support arrived in the 1.1.0 line (snapshot.4 shipped 26.1->26.2 shims with 188 verified class moves/renames plus 209 field redirects, harvested by scripts/harvest-mc-diff.py; verified on 26.2-rc-1 at 214/214). 1.1.0 stable shipped Fabric+NeoForge 26.2 jars with first-round Vulkan compatibility (prefer the OpenGL backend). By 1.2.0, 26.2 was a maintained target on all three loaders once Forge shipped a 65.x build, with 26.2 render-API stand-ins and the ColorCollection field consolidation covering the removed per-color statics.

26.x worldgen for structure mods (the Yung family)

1.2.0-snapshot.7 took worldgen-heavy mods from 'registers nothing' to structures generating in-world on a 26.2 dedicated server. The likely root cause was found (rewritten jars dropped ZIP directory entries, so Reflections/ClassGraph scanners found nothing). On top of that, byte-identity preservation, IntProvider/FloatProvider became-interface handling, jigsaw-generation bridges (JigsawBlockInfo unwrapping, canAttach re-typing, getHolder->get), and StructureProcessor superclass-rebase let YUNG's API + Better Dungeons register and generate (/locate + /place work). snapshot.2's ModDataMigrator had already fixed the strict-JSON worldgen cascade (one mod's lenient JSON aborting the shared registry load).

A 1.12.2 pre-1.13 Forge baseline

1.2.0 landed transform-level groundwork for the largest legacy mod library: a 117-entry 1.12.2->26.1 class-move table (snapshot.6) plus a 125-row 1.20.x-target variant (snapshot.8), a Block/Item Properties-constructor bridge, a 7,736-entry SRG member namespace (snapshot.8), the pre-1.13 FML lifecycle idiom, mcmod.info->mods.toml generation (snapshot.7), and load-layer stubs for removed Forge API types and registration events (rc.1). This gets a 1.12.2 mod scanned and past the first wall of NoClassDefFoundErrors (validated in-game on Forge 1.20.1), but getting mods to fully RUN is explicitly deferred to the future 1.4.0 line.

Broader pre-26.1 Fabric coverage (#55)

On a pre-26.1 host the intermediary->Mojang remap is correctly off and the legacy Fabric shims were authored in the wrong namespace, so old Fabric mods break on redesigned/removed APIs. Two engines address this: a generic FuzzyMethodResolver that indexes the host jar and scores rename candidates, and specific host-introspecting Pre<version><API>Bridge classes. 1.2.0-snapshot.3 finished the pre-1.17 model bridge (abstract model bases), and snapshot.8's live acceptance pass ran real 1.16.5 Serilum mods clean through a 1.20.1 Fabric server start, adding pre-1.19 Text, pre-1.17 Entity.onGround, and pre-1.20 Material bridges. The dead legacy MCP/Yarn shim prune is still ahead.

Self-hash authenticity and compatibility-DB / Probe tooling

Official builds embed a SHA-256 of Retromod's own classes; at startup the verifier reports VERIFIED on a match, an integrity/modification check (not cryptographic anti-tamper, since there is no secret key). The 1.1.0 line shipped the community Compatibility DB (ProtonDB-style reports plus the full 5,000-mod static audit) as the ground truth, and settings-screen authenticity status. 1.2.0-rc.1 added Probe, an upload-free in-browser compatibility pre-check backed by a database distilled from Retromod's rename/class-move tables (a stored DOM-XSS and jar-reader hardening were fixed in rc.2). Related engine tooling includes the descriptor-aware gap report and the fuzzy-resolver TSV for offline bridge authoring.

A note on measurement

Why there is no single “% compatible” here

The obvious headline would be a compatibility percentage from the top-5,000-mod audit. We rebuilt that audit against the current engine and deliberately did not publish one figure, because the comparison would mislead.

The old audit (May 2026)
Counted a class as resolved if Retromod’s own mapping contained the name, even when that name is a class 26.1 deleted. Optimistic by construction.
The current check
Counts a class as resolved only if it exists in the real 26.1 jar. Stricter, and honest about references that map onto removed classes.

Because the two use different definitions of “resolved,” the newer number reads lower even where nothing regressed. We verified this: the mods that look worse (ferrite-core, lithium) fail only on the 26.1 model-system rewrite (BakedModel, ModelResourceLocation and kin), which was removed and reshaped, not renamed. That gap was documented as deferred back in snapshot.2 and never bridged. It is a real, known limitation, not a regression.

So instead of a flattering number, this page lists the concrete capabilities that shipped and, below, the ones that did not.

Straight talk

What still doesn’t work

The honest column. These are carried openly in the changelog and docs.

won’t

Heavy coremod/rendering mods cannot be translated. Create (ships Flywheel, a custom GL renderer plus coremods), Flywheel, and Veil are on the 'Mods That Can't Be Translated' list; they fail with coremod/getLoadingModList/VerifyError/CancellationException-teardown symptoms regardless of metadata fixes, and are not chased as transform bugs.

limit

Deep Forge->NeoForge migrations are deferred. 1.2.0 landed the construct+register spine (verified on small Macaw's content mods), but the long tail of net.minecraftforge.* APIs beyond that verified path (data gen, deeper event coverage, complex mods, reshaped systems like the TickEvent phase redesign and the capability system) is still a Sinytra-Connector-scale follow-up.

limit

The 26.1 model-system rewrite is not yet bridged for old Fabric render/model mods. BakedModel, ModelResourceLocation, ItemRenderer and the related 26.1 render-pipeline classes were flagged (snapshot.2) as removed-and-reshaped, not just renamed, so a stub would pass a static check but crash in game; those old Fabric render/model mods still show gaps and need the feature rewritten rather than redirected.

limit

1.12.2 mods load but do not fully run in-game yet. The class-move tables, ctor bridge, SRG member namespace, FML lifecycle layer, and load stubs are transform-level building blocks (validated in-game only as far as scanning/constructing past the first wall on Forge 1.20.1); past that each mod hits a different removed system (commands, model events, GUIs, networking, entity AI), which is explicitly the future 1.4.0 line's job.

limit

The 26.2 render-API stand-ins only make render-referencing mods LOAD, not fully render. The endBatch/frame-submission family through the new SubmitNodeCollector pipeline is staged as a follow-up (warns once and drops the batch), and the load path is transform-level (RenderBufferSyntheticsTest), not yet in-game verified.

limit

Many reflective API bridges are implemented pending in-game verification and fail soft. Numerous Fabric event/SAM bridges (item-group events, ServerWorldEvents, EntityModelLayerRegistry, CPN v1, ClientTickCallback) are bytecode-tested but their reflective event wiring still needs an in-game check and falls back to inert/no-op rather than crashing if it can't wire up.

won’t

Some mods are confirmed genuinely un-translatable by a mechanical transformer. True Darkness (#68): LightTexture was deleted and redesigned into a GPU/UBO Lightmap, so the mixin's @Shadow has no equivalent (the strip stays). YUNG's Better End Island works on 26.1 but is a deep 26.2 EnderDragonFight redesign, not a re-signature case.

limit

The Yung's API worldgen mixins (BeardifierMixin/NoiseChunkMixin) remain soft-failed (stripped) pending verification. Research confirmed both are mechanically repairable, but re-enabling interdependent worldgen mixins needs a headless-server worldgen pass first, since a mis-repair cascades to all chunk generation.

limit

The MixinExtras inner-target rewriter is incomplete for the @Local slot re-resolution case (the roadmap's original Deeper and Darker @WrapOperation + @Local Level example); the current jars did not exercise it, so it is deferred until a mod does.

won’t

Retromod will not translate arbitrary OpenGL bytecode to Vulkan. The 26.2-window fix just prefers the OpenGL backend (Tier 0) and neutralizes removed render-state calls (Tier 2, inert soft-fail). Raw-GL / custom-shader / immediate-mode mods are a hard boundary, and MC 26.3 removing OpenGL entirely is a designed-not-built future problem.

limit

The self-hash authenticity check is an integrity/modification check, not cryptographic anti-tamper. There is no secret key, so a determined attacker can recompute it; for real verification users compare the jar's SHA-256 against the value published on the releases page. The check is informational and never a gate (MIT).

limit

The mods/Retromod/ CurseForge-override locator is currently unavailable on Forge and NeoForge. Declaring the locator SPI made the loader claim Retromod's jar as an early service and skip its @Mod (removed on Forge in snapshot.5, NeoForge in snapshot.7); a locator-only stub jar is the planned follow-up, and until then old mods go through retromod-input/ or directly in mods/.

limit

The bundled intermediary-to-mojang table is 1.21.4-based, so it carries stale names for any method renamed between 1.21.4 and 26.1 (which breaks Fabric overrides on a 26.1+ host); the AER tooltip bug was one instance and more such renames may need the same correction as they surface. Per-era intermediary data for pre-26.1 mods is still an open dependency for authoring correct old-mod bridges.
